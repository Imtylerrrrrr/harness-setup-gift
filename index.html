AI 코딩 세팅 가이드 · 개발자 꿈나무용

같은 AI를 쓰는데,
결과는 왜
사람마다 다를까

AI에게 코딩을 시키는 것까지는 누구나 합니다. 차이는 세팅에서 갈립니다.
규칙 파일 하나, 스킬 하나, 연결 하나 — 이 작은 세팅들이 AI를 "말 잘 듣는 팀원"으로 바꿔 놓습니다.

이 문서는 그 세팅을 딱 3층으로 정리한 가이드입니다. 터미널을 잘 몰라도 따라올 수 있게 만들었습니다.

함께 보기: 이 저장소의 examples/(실제 세팅 파일 모양) · exercises/(따라 하기 3편)

First, The Concept

하네스 — AI에게
손과 발을 다는 것

하네스(harness) → 말에 씌우는 마구(馬具). AI를 실제로 "일하게 장착"시키는 장치라는 뜻입니다.

언어 모델 자체는 말만 할 수 있는 천재입니다. 설명은 해주지만 직접 하지는 못합니다. 여기에 파일을 만지는 손, 명령을 실행하는 발, 결과를 확인하는 눈을 달아주는 도구가 하네스입니다. Claude Code가 대표적입니다.

그냥 AI (채팅창)

  • 답변(글)만 준다
  • 코드를 복사해서 내가 붙여넣어야 한다
  • 실행해보고 에러를 다시 내가 전달한다
  • 대화가 끝나면 다 잊는다

하네스를 단 AI

  • 직접 파일을 만들고 고친다
  • 직접 실행하고 에러도 직접 본다
  • 될 때까지 수집→행동→검증을 반복한다
  • 규칙·기억을 파일로 이어간다

좋은 소식 — 하네스는 이미 달려 있다

Claude Code나 Codex 같은 코딩 에이전트를 써봤다면, 이미 최상급 하네스를 쓰고 있는 것입니다. 반복 루프, 파일·실행 도구, 안전장치(위험한 일은 먼저 허락받기), 기억 관리까지 전부 완성품으로 들어 있습니다. 우리가 새로 만들 부품은 없습니다.

그런데 완성품이 모르는 게 딱 하나 있다

바로 나입니다. 내 코딩 스타일, 내 프로젝트 구조, 내가 매번 반복하는 작업 — 이건 어떤 회사도 미리 넣어줄 수 없습니다. 세팅이란 완성품 하네스에게 "나"를 가르치는 일입니다. 게임도 기본 설정으로 플레이는 되지만, 잘하는 사람은 자기 감도로 맞추고 시작하는 것과 같습니다.

프롬프트는 그 대화에서 증발하지만,
세팅은 파일로 남아 계속 일한다.
— 같은 말을 반복할 필요가 없어지고, 결과의 들쭉날쭉함이 줄어듭니다. 그게 전부지만, 매일 쓰는 도구에서는 그 차이가 꽤 큽니다.
The One Metaphor

세팅은 렌즈를
갈아끼우는 일이다

광각 렌즈 → 넓게 전부 담는 렌즈 / 매크로 렌즈 → 아주 가까이 확대하는 렌즈. 카메라 비유입니다.

세팅 파일은 어디에 두느냐에 따라 미치는 범위가 달라집니다. 내 컴퓨터 전체에 거는 세팅, 프로젝트 하나에 거는 세팅, 폴더 하나에 거는 세팅 — 딱 세 층입니다. 이 문서 전체가 이 그림 하나입니다.

렌즈를 내릴수록 좁고 깊어집니다. 겹치면 안쪽(더 구체적인) 규칙이 이깁니다.

이미 아는 원리입니다 — 지역변수가 전역변수를 가리는 것, CSS에서 더 구체적인 선택자가 이기는 것과 같습니다. 넓은 세팅은 한 번만, 좁은 세팅은 공들이는 곳에만.

The Ingredients

층에 얹을 재료는
딱 5개만 알면 된다

세팅이라고 해서 종류가 많지 않습니다. 이 5개를 어느 렌즈(층)에 두느냐가 다음 세 섹션의 전부입니다.

규칙 · 항상 읽음

📋 CLAUDE.md

AI가 매번 자동으로 읽는 업무 매뉴얼. "이 규칙은 항상 지켜"를 적는 곳.

예: 코딩 스타일 · 프로젝트 구조
절차 · 필요할 때만

📖 Skills

반복 작업의 레시피북. 그 작업이 올 때만 펼쳐 읽어서 가볍다.

예: 문서 만들기 · 세트 추가 절차
묶음 · 설치해서 씀

🧩 Plugins

남이 잘 만든 세팅 묶음을 장터에서 설치. 직접 안 만들어도 된다.

예: /plugin 명령으로 설치
연결 · 외부 도구

🔌 MCP

AI를 바깥 도구와 연결하는 표준 규격. 문서 검색, 게임 엔진까지.

예: 최신 문서 · Unity 에디터
강제 · 자동 발동

🚪 Hooks

정해진 순간(예: 파일을 고친 직후)에 무조건 자동 실행되는 문지기. 앞의 넷이 "부탁"이라면 Hooks는 "강제" — CLAUDE.md의 규칙은 AI가 가끔 어기지만, Hook은 어길 수가 없다. 그래서 반드시 지켜져야 하는 것에만 쓴다.

예: 코드 고칠 때마다 자동 검사·포맷

다섯 개 다 "파일 하나 또는 명령 하나"로 켜집니다. 이제 층별로 — 🔭 광각부터 갑니다.

Lens 1 · Wide

🔭 광각 — 한 번 깔면
모든 프로젝트에 적용된다

~/.claude/ → 내 홈 폴더 안의 .claude 폴더. 여기 둔 세팅은 이 컴퓨터의 모든 프로젝트에서 적용됩니다.

어느 프로젝트에서든 똑같이 통하는 것만 여기 둡니다. 내 스타일 규칙(CLAUDE.md)과 어디서나 쓰는 범용 스킬(문서 만들기 같은 것) — 딱 그 두 종류입니다.

대표 예시 — 카파시(Karpathy) 4원칙 플러그인

AI 연구자 안드레 카파시가 2026년 1월, AI 코딩의 실패 패턴을 정리해 올렸습니다 — 멋대로 가정하고, 과하게 복잡하게 만들고, 옆 코드까지 건드리는 문제들. 이를 규칙 파일로 만든 플러그인이 GitHub 스타 10만을 넘겼습니다. 명령 두 줄이면 모든 프로젝트에 깔립니다.

# Claude Code 안에서 그대로 입력
/plugin marketplace add forrestchang/andrej-karpathy-skills
/plugin install andrej-karpathy-skills@karpathy-skills

참고: 카파시 본인이 만든 것은 아니고, 그의 글에서 영감을 받아 만들어진 것입니다.

원칙 1

코딩 전에 생각

불확실하면 가정을 말하고 물어본다. 멋대로 정하지 않는다.

원칙 2

단순하게

요청받은 것만. "나중에 쓸지도 모르는" 코드는 만들지 않는다.

원칙 3

수술하듯 고치기

부탁받은 부분만 건드린다. 옆 코드 "개선" 금지.

원칙 4

목표 중심

"고쳐줘"를 "테스트 통과"처럼 확인 가능한 목표로 바꾼다.

이 4원칙의 축약판 CLAUDE.md가 examples/1-global/에 있습니다. 복사해서 내 것으로 고쳐 써도 됩니다.

Lens 2 · Standard

🎯 표준 — 이 프로젝트에
맞는 것만 담는다

프로젝트 폴더 안의 .claude/와 .mcp.json. 이 프로젝트에서 일할 때만 적용됩니다.

프로젝트 층에 얹는 것은 세 가지입니다.

1

프로젝트 규칙 — CLAUDE.md

그리고 — 이렇게 만든 스킬은 공개하면 남의 세팅도 됩니다. 아래 두 개는 제가 직접 만들어 공개한 개인 스킬입니다. 유명한 검증된 도구가 아니니 바로 설치하기보다는, SKILL.md가 실제로 어떻게 생겼는지 구경하는 용도로 열어보길 권합니다.

여럿이 AI 에이전트와 함께 일할 때 PR·커밋·브랜치 규칙을 인터뷰 4문항으로 세팅해주는 스킬. 규칙을 문서가 아니라 GitHub 설정으로 강제하는 게 특징입니다. — github.com/Imtylerrrrrr/TK

둘 다 처음엔 "자기한테 필요해서" 만든 것입니다. 반복되는 불편 → 스킬 → 공개. 이 순서는 누구나 밟을 수 있습니다.

Lens 3 · Macro

🔬 매크로 — 공들이는
폴더에만 한 겹 더

폴더 안에 CLAUDE.md를 한 장 더 두면, AI가 그 폴더의 파일을 다룰 때만 추가로 읽습니다.

모든 폴더에 할 필요는 전혀 없습니다. 유독 아끼는 부분, 유독 실수가 잦은 부분 — 거기에만 씁니다. 이를테면 이런 곳입니다.

데이터 폴더 — "여기 데이터는 출처 주석 필수, 수정하면 변경 기록 한 줄" · 핵심 로직 폴더 — "이 폴더는 테스트 없이 수정 금지" · 발표 자료 폴더 — "여기 파일은 문체 통일, 오타 이중 확인"

examples/2-project/3-folder/가 딱 이 예시입니다. 고난도 세트 폴더에만 "단어 20개 이상, 출처 주석 필수" 같은 더 엄격한 규칙이 한 겹 얹혀 있습니다.

넓은 렌즈는 한 번만,
좁은 렌즈는 공들이는 곳에만.
— 3층을 다 채우는 게 목표가 아닙니다. 글로벌 하나 + 프로젝트 하나면 대부분 충분하고, 매크로는 필요해질 때 얹는 것입니다.
Common Pitfalls

세팅에도 함정이 있다
— 딱 3가지만 조심

1

다 CLAUDE.md에 욱여넣기

CLAUDE.md는 매번 읽힙니다. 길어질수록 AI의 머리(컨텍스트)가 무거워집니다. 항상 필요한 규칙만 남기고, 가끔 필요한 절차는 스킬로 빼는 것이 정답입니다.

2

안 쓰는 세팅 수집하기

플러그인과 MCP를 잔뜩 깔아두는 것 자체가 실력이 되지는 않습니다. 오히려 충돌하고 느려집니다. 지금 하는 작업에 쓰는 것만 켜두고, 안 쓰면 지웁니다.

3

세팅을 믿고 검증을 건너뛰기

규칙을 적어놨다고 AI가 항상 지키는 것은 아닙니다. 결과를 실행해서 확인하는 습관은 세팅이 대신해주지 않습니다. 세팅은 확률을 높일 뿐, 확인은 내 몫입니다.

황금률 한 줄

가장 좋은 초기 세팅은, 최소한의 세팅입니다. CLAUDE.md 한 장에서 시작 → 부족함을 느끼면 스킬 → 그래도 부족하면 MCP. 미리 다 깔아두는 게 아니라 부족함을 느낀 다음에 하나씩 얹는 것 — 이 순서를 지키는 것이 고수의 세팅입니다.

🔁 Loop 엔지니어링 — 프롬프트의 다음 단계

Claude Code를 만든 개발자 보리스 체르니는 최근 이렇게 말했습니다. "이제 나는 Claude에 직접 프롬프트하지 않는다. 루프가 Claude에 프롬프트한다. 내 일은 루프를 쓰는 것." 사람이 시키고 기다리는 대신, 계획 → 실행 → 검증 → 다음 계획을 스스로 도는 자동 사이클을 설계하는 것 — 2026년 여름, 이 방식에 "loop engineering"이라는 이름이 붙었습니다. 세팅(이 가이드)이 익숙해진 다음에 열리는 문입니다.
